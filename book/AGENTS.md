# AGENTS.md — the book's format, build, and conventions

*3D Printing Production Software* renders from markdown source into a small static HTML site by
`book/build_book_html.py` (stdlib-only, no dependencies). This file is what a new contributor or agent
needs to build and edit the book correctly. It also specifies the **index-annotation method** — a
proposed way to point index entries at curated concept anchors instead of raw text occurrences.

Companion: `book/README.md` is the short author's-eye view; this file is the full format+build reference
and the design record for the annotation work.

---

## 1. How to build

- **Canonical:** from the catalogue root, `python3 catalog.py build`. It renders every catalogue entry,
  regenerates the census views, and — as one of its stages — invokes the book renderer. It also runs a
  **BLOCKING orphan-reachability gate** over the book pages (a page rendered but linked from nowhere
  fails the build).
- **Book only:** `python3 book/build_book_html.py` renders just the book (stdlib-only, runs on a fresh
  checkout with nothing installed).
- **Never hand-edit the `.html`.** Every page is generated and overwritten on the next build. Edit the
  markdown source, then rebuild.

**What the build emits, all flat in `book/`:**

- One `<slug>.html` per chapter (e.g. `3.2-models-and-the-semantic-gap.html`), where `<slug>` is the
  source file's stem.
- `index.html` — the landing / table-of-contents page.
- `appendix-a-agent.html`, `appendix-b-models-bridge.html`, `appendix-c-product.html` — the generated
  Gang-of-Four appendix, one page per catalogue role.
- `book-index.html` — the autogenerated term index.
- `catalogue-figure.html` — a rewired copy of the catalogue's clickable mechanism-map figure (its chips
  re-pointed to the in-book appendix patterns).

## 2. Source layout — the Part/Chapter hierarchy

The **directory tree encodes the ordering**. A chapter lives at `part<N>/<N>.<M>-<slug>.md`, so the part
number and chapter number are explicit in the path. The build walks the tree, derives `PART.CHAPTER`
from the path, and reads the title metadata from each file.

```
book/
  frontmatter/    0.1-preface.md, 0.2-acknowledgments.md   (Part 0 — Front Matter)
  part1/          1.1-…, 1.2-…                              (Part 1 — The Context)
  part2/          2.1-…, 2.2-…                              (Part 2 — The Mindset)
  part3/          3.1-… … 3.4-…                             (Part 3 — The Governed Engineering Environment)
  part4/          4.1-… … 4.6-…                             (Part 4 — The Model Zoo)
  part5/          5.1-… … 5.5-…                             (Part 5 — Putting It to Work)
  backmatter/     6.1-conclusion.md                         (Part 6 — Back Matter)
  data/metrics.json          # headline numbers, referenced from prose via {{token}}
  assets/                     # figure assets (inline SVGs, rasters)
  appendix-fills/<role>/<slug>.md   # per-entry Structure + Sample Code fills for the appendix
  index-terms.md             # curated supplement to the term index
```

- **Part numbers:** front matter is Part 0, the numbered Parts are 1–5, back matter is Part 6. Appendix
  Parts (A/B/C) are numbered after these.
- **The filename's leading digit must match its directory** — a `2.1-…md` under `part1/` fails the build
  loud (guards a misfiled chapter).
- **Chapters sort by `(part, chapter)`.** A stray non-`<N>.<M>-` file in a part dir is skipped, not
  rendered.

### Chapter metadata

Each chapter carries two HTML-comment fields, read by the build:

- `<!-- part-title: … -->` — the Part's display title.
- `<!-- chapter-title: … -->` — the chapter's title (used in the header, TOC, and pager).

The leading `# …` H1 is **dropped on render** (the header comes from metadata), so keep or change it
freely. Front and back matter render without a "Chapter N" kicker.

### Per-Part epigraphs

The first chapter of each **numbered** Part opens with an epigraph. These are not in the markdown — they
live in `_PART_EPIGRAPHS` in `build_book_html.py`, one `(quote, attribution)` pair per Part. The Macbeth
(Part 2) and Ecclesiastes (Part 5) quotations are verbatim from the source memoir; the Part 1, Part 3, and
Part 4 (the George Box "all models are wrong" line for the Model Zoo) openers are editor-swappable
candidates.

### Copyright footer

Every page footer carries `© James C. Davis, 2026–present` (the `COPYRIGHT` constant). It is emitted by
the build; do not add it to prose.

## 3. Authoring conventions and directives

The renderer understands a small markdown subset plus a few directives. All directives are HTML comments
so they stay invisible if the markdown is read raw.

- **Figures — `<!-- figure: <path> | <caption> -->`.** `<path>` is relative to `book/`. An `.svg` asset
  is **inlined** (its own `<title>`/`<desc>`/`aria-*` survive, and there is no external request to 404);
  any other extension is wrapped in `<img alt="<caption>">`. A missing asset **fails the build**.
- **Interactive figures — `<!-- figure-iframe: <path> | <caption> | <a11y-title> -->`.** Renders a
  `<figure>` with an `<iframe>` preview plus a through-link. Used to surface a self-contained catalogue
  figure page (whose internal links are book-relative) live, without splicing its styles/scripts inline.
  The `<iframe>` carries a required accessibility `title`. A missing target page fails the build. This is
  how the appendix embeds the rewired mechanism map.
- **Metrics tokens — `{{token}}`.** Recurring numbers (weeks, LoC, costs) live in `data/metrics.json`.
  Reference them from prose as `{{token}}`; the build substitutes the value. An **unknown token fails
  loud** — a mistyped placeholder stops the build rather than shipping `{{typo}}`. Keys prefixed with `_`
  in the JSON are notes, not tokens. **Edit the number in the JSON, never in the prose.**
- **Headings.** `#`, `##`, `###`. A trailing `{#slug}` on a heading sets a stable `id` anchor and is
  stripped from the visible text (the appendix uses this so the mechanism-map figure can deep-link each
  pattern).
- **Gap markers.** A block starting `[FILL IN: …]` or `[MORE CHAPTERS FOLLOW: …]` renders as a visible
  callout box — the draft's to-do markers, deliberately shown.
- **Also supported:** unordered lists (`- `), blockquotes (`> `), fenced code blocks. A ` ```mermaid `
  fence renders as a live diagram (Mermaid runtime is loaded only on pages that need it — the appendix).
  Inline: `**bold**`, `*italic*`, `[text](url)`, and `[[slug|text]]` abstraction citations (link into the
  catalogue's glossary).

## 4. The appendix system — GoF patterns from the catalogue

The three appendix pages are **generated at build time from the catalogue entries**, so they cannot drift
from the catalogue. Each catalogue entry (`<role>/<family>/<mechanism>.md`, roles `agent` /
`models-bridge` / `product`) is re-projected into the classic Gang-of-Four layout: Intent · Motivation ·
Applicability · Structure · Sample Code · Consequences · Known Uses · Related Patterns.

- **One appendix Part per role** — Appendix A (Agent), B (Models-bridge), C (Product). Each opens with a
  blurb plus the embedded rewired mechanism-map figure, then lists patterns grouped by family.
- **Projected slots** (Motivation, Applicability, Consequences, Known Uses, Related Patterns) are read
  straight from the catalogue entry's sections. Relative `.md` links in the entry are rewritten so they
  resolve from `book/*.html`.
- **Per-entry fills** supply the two slots the catalogue entry does not carry: **Structure** (a
  ` ```mermaid ` diagram + an accessible-description line) and **Sample Code** (framing prose + a code
  block, or a prose "no sample code" note for a policy control). Fills live at
  `appendix-fills/<role>/<slug>.md`, **keyed by the catalogue entry's slug** (the fill's filename stem
  must match the entry stem). Inside a fill, the build extracts the `### Structure` and `### Sample Code`
  sections. A missing fill or missing slot renders a visible `[FILL IN: …]` TODO.
- **Adding a fill:** create `appendix-fills/<role>/<slug>.md` with a `### Structure` and a `### Sample
  Code` section. See `appendix-fills/product/pdf-model.md` as the reference exemplar.

## 5. The index system — autogenerated term index

`book-index.html` is generated by merging term sources, occurrence-scanning the prose, and emitting an
alphabetized index. It is a **soft, best-effort** index: a term that never appears in the prose is
dropped, so the index lists only findable terms.

**Term sources (merged, de-duplicated):**

1. **The self-communicate lexicon** — the catalogue's house vocabulary (the bold first-column terms of
   the lexicon table), read at build time. Covers the appendix + technical terms automatically.
2. **`book/index-terms.md`** — the book's own curated concepts and proper nouns (bulleted; may carry an
   `@ch..` primary-chapter hint that is stripped for matching).

**Scan and rank:** for each candidate term, the build scans every chapter + appendix page. A page where
the term appears in a **heading** ranks above a body-only mention; ties break on reading order.
References are **capped per term** (currently 4) so the index reads curated, not a frequency dump.
Terms shorter than 3 characters are skipped. A `/`-run in a term ("runbook / playbook") matches either
alternative; a trailing `(qualifier)` is dropped for matching.

## 6. Index annotations — curated concept entries

**The need.** The current index is occurrence-driven: it lists the pages where a term *appears*, ranked
by a heading heuristic. That answers "where is this word?" but not "where is this concept *defined*, and
where is it *exemplified*?" For a concept like the semantic gap, a reader wants the defining paragraph
first, then its worked examples — not a ranked list of every mention. The method below lets an author
place **curated anchors** at the canonical paragraph and its examples, and have the index emit a
concept-shaped entry that leads with them.

This is a **design** for the annotation method. Implementing the tags in the renderer, and authoring the
tags across the book, are follow-ups (see §6.5) — not part of this document's change.

### 6.1 Annotation syntax

Two HTML-comment tags, placed on their own line at (or immediately before) the canonical paragraph or
heading. HTML comments match the book's existing directive style, stay invisible if the markdown is read
raw, and survive the block splitter.

- `<!-- index-def: <concept-slug> -->` — marks the **definition** of a concept. At most one per concept
  across the whole book (a second is a build-loud error — a concept has one canonical definition).
- `<!-- index-example: <concept-slug> -->` — marks an **example** of a concept. Any number, across any
  chapters.

`<concept-slug>` is a stable kebab-case id (e.g. `semantic-gap`, `loop-engineering`). The renderer turns
each tag into a stable anchor on the enclosing block: `id="idx-def-<slug>"` for a definition,
`id="idx-ex-<slug>-<n>"` for the n-th example (numbered in reading order). The tag itself renders to
nothing visible — only the anchor is emitted on the following block.

**Concept registry.** The set of concept slugs and their display names lives in a small curated list —
extend `book/index-terms.md` with a `concept:` line form, e.g. `- concept: semantic-gap | The semantic
gap`, so the display name is authored once and not scraped from prose. A tag whose slug is not in the
registry is a build-loud error (catches a typo before it silently drops the concept).

### 6.2 How the index generator consumes them

At build time, before the occurrence scan, the generator walks every chapter + appendix page for the two
tags and records, per concept slug: the one definition anchor `(page, id)` and the ordered list of
example anchors `[(page, id), …]`. It then emits a **curated concept entry** in this shape:

```
SEMANTIC GAP
  definition of: 3.2
  examples of:   3.2  3.3  4.5
```

where each locator is a link to the specific anchor (`3.2-…html#idx-def-semantic-gap`, etc.), using the
same short page label (`N.M` / `Appendix A` / `Preface`) the occurrence index already uses.

**Composition with the occurrence scan (curated first, occurrence as fallback):**

- A concept that has at least one `index-def` or `index-example` tag renders as a **curated entry** (the
  `definition of:` / `examples of:` shape). Its curated anchors replace the occurrence-scan refs for that
  term — the author has said where the significant sites are.
- A term with **no** curated tags falls back to the existing **occurrence entry** (the capped, ranked
  page list). Nothing regresses: the annotation method is purely additive.
- The two entry kinds interleave alphabetically in one index. A curated entry is visually distinguished
  (the `definition of:` / `examples of:` sub-lines) from a bare occurrence entry (a comma-run of page
  links).
- If a concept slug also matches a lexicon/`index-terms` display term, the curated entry **wins** (the
  occurrence entry for that term is suppressed, so a concept is not listed twice).

### 6.3 Worked examples — "Observability" and "Quality"

These two are the sharpest illustration of why a curated entry beats an occurrence scan. Each is a bare
word that appears **many times** across the book — in prose, in headings, in the appendix. An occurrence
scan gives the reader a ranked list of page hits and no way to tell the defining paragraph from the
twentieth passing mention. What the reader actually wants is the one paragraph that **defines** the
concept and the handful that **exemplify** it. The annotation tags mark exactly those.

**Observability.** Define it once (say, in the governed-environment or lifecycles material), then tag its
worked examples where the book discusses the event bus, the cron-alerts gate, and diagnosability-driven
signals.

At its defining paragraph:

```markdown
<!-- index-def: observability -->
**Observability** is the property that a running system exposes enough signal to explain
what it did and to catch what went wrong — the sensors that make a failure legible before
it becomes an incident. …
```

At two example paragraphs (the typed event bus; the cron-alerts gate):

```markdown
<!-- index-example: observability -->
The typed event bus is the substrate here: every mechanism that emits a topic owns the
observability for it end-to-end — the baseline-healthy shape, what-looks-wrong, and the
playbook entry ship with the emitter …
```

```markdown
<!-- index-example: observability -->
The cron-alerts gate turns an unresolved high-severity alert into a hard stop on new
work-dispatch — observability that doesn't merely report but blocks …
```

Register the concept once in `book/index-terms.md`:

```markdown
- concept: observability | Observability
```

**Resulting index entry** in `book-index.html`:

```
OBSERVABILITY
  definition of: 3.3
  examples of:   3.5  4.5
```

where `3.3` under `definition of:` links to `#idx-def-observability`, and each `examples of:` locator
links to its `#idx-ex-observability-<n>` anchor. The old occurrence entry for "observability" — which
would have listed a long comma-run of undifferentiated page hits — is suppressed in favour of this
curated one.

**Quality** follows the same shape: one `<!-- index-def: quality -->` at its defining paragraph, then
`<!-- index-example: quality -->` at the paragraphs that exemplify it across **correctness**,
**conformance**, and the **testing-onion** material — yielding `QUALITY` / `definition of: <link>` /
`examples of: <link> <link> <link>`.

**Seed list.** These two, plus concepts like **the semantic gap** (defined in 3.2), **loop engineering**,
and **the drift gate**, are the seed set the later walk-the-book pass (§6.5) will expand into the full
concept registry.

### 6.4 Why this shape

- **Curated over scraped.** The author names the definition and the examples; the index stops guessing
  which mention matters via a heading heuristic.
- **Additive and safe.** Untagged terms keep their occurrence entry, so the book indexes fully before any
  tag is authored, and improves concept-by-concept as tags land.
- **Stable anchors.** The `idx-def-`/`idx-ex-` ids are deep-linkable and survive re-rendering, so
  external references and the concept links stay valid.

### 6.5 Follow-ups (NOT part of this change)

1. **Implement the tags in the renderer.** Extend `build_book_html.py`'s index generator to (a) recognize
   `index-def` / `index-example` comments in `md_to_html` and emit the anchors; (b) collect them across
   pages; (c) read the `concept:` registry lines from `index-terms.md`; (d) emit curated concept entries
   and suppress the matching occurrence entries; (e) fail loud on a duplicate `index-def`, an unregistered
   slug, or an example with no definition.
2. **Walk-the-book authoring pass.** Once the tags are implemented, do a pass over the chapters to (a)
   enumerate the concept list (the `index-terms.md` concepts are the seed), and (b) place `index-def` at
   each concept's defining paragraph and `index-example` at its worked examples.

## 7. Quick checklist for editing the book

- Edit markdown under `part<N>/`, `frontmatter/`, `backmatter/` — never the `.html`.
- New recurring number → `data/metrics.json`, referenced as `{{token}}`.
- New figure → drop the asset in `assets/`, reference with `<!-- figure: assets/<file> | caption -->`.
- New appendix Structure/Sample Code → `appendix-fills/<role>/<slug>.md`, keyed by the entry slug.
- New index concept → register in `index-terms.md`; annotate per §6 once the tags are implemented.
- Rebuild with `python3 catalog.py build` (runs the reachability gate); commit both the source and the
  generated HTML.
