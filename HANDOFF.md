## 1. What this repo is

A **standalone public repo** — a pattern catalogue of the **governance mechanisms** for
*governance-centric agentic software engineering*: the controls that let a fleet of AI coding agents stay
productive while bounding their failures. Each mechanism is a Gang-of-Four-style design-pattern writeup.

- Distilled from a 12-week first-person **case study** (the paper: `https://arxiv.org/pdf/2607.01087`,
  *Cheap Code, Costly Judgment*) of building the **ScholAccess** document-accessibility system with
  frontier coding agents. The live system: `https://scholaccess.com`.
- **Published** as a static GitHub Pages site: `https://davisjam.github.io/agent-governance-mechanisms/`.
- **Embedded** in a private parent repo (`ada-tool`) as a **git submodule** at
  `talks-and-notes/governance-catalog`. This repo is the origin `git@github.com:davisjam/agent-governance-mechanisms.git`.
- The parent repo (`ada-tool`) is the real system these mechanisms were extracted from; it is **private
  and NOT shipped**. That fact drives the governing principle below.

## 2. Governing principle (the load-bearing rule)

**This repo must be interpretable by an independent Claude** — an agent with no access to `ada-tool`, its
source, docs, tools, or CLAUDE.md rule numbers. Every entry must stand on its own. The full rule and the
test are in [`CLAUDE.md`](CLAUDE.md) §"Core principle: this repo must be interpretable by an independent
Claude". **This is the largest open work item** — see §9.

## 5. `catalog.py` — the tool (stdlib-only, no pip)

| Command | Does |
|---|---|
| `python3 catalog.py validate` | schema + INDEX-consistency + link-integrity + summary checks; exit 1 on any violation |
| `python3 catalog.py build` | render every `.md` → sibling `.html`; regenerate `index.html` + `catalogue-views.html` |
| `python3 catalog.py query [--role --family --form --enf] [--json]` | list/filter entries |
| `python3 catalog.py summaries [--json]` | dump role/family/entry summaries |
| `python3 catalog.py deploy local [--port N]` | validate + build + serve at `http://127.0.0.1:8137/` |
| `python3 catalog.py deploy github [-m msg]` | validate + build + commit + push (triggers Pages) |
| `python3 catalog.py install-hooks` | `git config core.hooksPath hooks` (auto-regen on commit) |

`catalog.py` contains a **compact, dependency-free markdown→HTML renderer** — it handles headers, tables,
lists, code spans/fences, bold/italic, links (rewrites `.md`→`.html`). It is deliberately not a full
markdown engine: **avoid `<br>` and bold-with-nested-italic (`**a *b* c**`) in `.md`** — they don't render
correctly. (BSD `sed` lacks `\b`; use `perl -pi -e` for word-boundary edits.)

## 6. Workflow to make and ship a change

1. Edit the `.md`.
2. `python3 catalog.py validate` (must be **0 issues**) → `python3 catalog.py build`.
3. Preview: `python3 catalog.py deploy local` → open `http://127.0.0.1:8137/`.
4. Publish: `python3 catalog.py deploy github -m "..."` (commits here + pushes → Pages deploys on push).
5. **Bump the parent pointer.** In `ada-tool` (the parent), the submodule gitlink must be advanced to the
   new public HEAD. Parent policy forbids the orchestrator committing on `main` directly, so this is done
   by a **commit-slave** sub-agent. The canonical commit-slave brief:
   - `export ADA_TOOL_AGENT_ROLE=commit-slave`
   - `git add talks-and-notes/governance-catalog` (ONLY the gitlink)
   - `python3 tools/agents/agent_commit.py --of 1/1 -m "chore(governance-catalog): bump submodule -> <sha> (...)"`
   - This is a parent-repo mechanic; from *inside* this repo you only ever `deploy github`.

The tracked `hooks/pre-commit` (install once via `catalog.py install-hooks`) re-runs validate+build+stage
on every commit, so committed HTML can never drift from the `.md`.

## 7. Schema (what `validate` enforces)

Each entry `<role>/<family>/<control>.md`:

- `# Title`, then `**Intent** — …`.
- A **metadata card** — exactly these rows, in this order:
  ```
  | | |
  |---|---|
  | Summary | <=100-char one-liner (shown as page subtitle + census column) |
  | Target | Agent|Bridge|Product · **<Family>** |
  | Form | `<one of nine>` |
  | Enforcement | **Hard**|**Soft**|**Soft·Hard** (…) |
  ```
  *(Novelty, Real artifact, Governing rule(s) were removed from the schema; Summary is first.)*
- **Form** ∈ {`typed-ir`, `validation`, `repair-vocab`, `agent-output`, `bounded-service`, `regression`,
  `quality-gate`, `observability`, `audit-trail`}. **Enforcement class** ∈ {Hard, Soft, Soft·Hard}.
- Seven `##` sections, in order: `Motivation` · `Why it's not just …` · `Mechanism` · `Prerequisites` ·
  `Consequences & costs` · `Known uses` · `Related controls`.
- `Related controls` bullets carry a **relationship tag** (Counterpart / Enabler / Layer / Consumer /
  Bridge / See also) and link other entries via relative `../<role>/<family>/<control>.md`.
- **INDEX.md** has one row per entry; its `Form` and `Enforcement` **must match** the entry's card
  (validated). INDEX row shape: `| ✅ | <Control name> | \`<form>\` | <Enf> | [<file>.md](<path>) |`.
- The umbrella `README.md` and the three role READMEs each carry a `<!-- summary: … -->` comment (used by
  `summaries`/tooltips and checked by validate).

## 10. Locked decisions (don't re-litigate)

- `catalog.py` stays **stdlib-only** (clone-and-run; no pip). It is the single source that builds the site.
- Fonts: **Source Serif 4** (headings) + **Source Sans 3** (body), from Google Fonts. Layout:
  `main { width:94vw; max-width:1320px }`; prose classes capped ~840px for readability; census/grids/
  figure use full width.
- The catalogue is **representative (~53 patterns), not exhaustive** of every lint/gate/test.
- The **`product/` role is flagged ScholAccess-specific** (a note renders above it in the census).
- Redaction: the product name is scrubbed everywhere; **scholaccess.com** and the **paper** are the two
  deliberate outward links.
- Schema slimmed to **Summary · Target · Form · Enforcement** (Summary first); Novelty / Real artifact /
  Governing rule(s) removed. Census shows **Control · Summary · Enforcement** (no Form column, no ★, no
  hover tooltip). The event-bus mechanism is titled **"Orchestrator-as-reactor over an event bus"**.
- The two newest mechanisms (both in the governance-doc family): **Operational playbooks** and **Epic &
  design-doc templates**.
